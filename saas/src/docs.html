OHD SaaS — API

Account · recovery · OIDC linking · plan · payments. v1 surface, JSON/REST over HTTPS. Base URL: https://api.ohd.dev
Auth Account Recovery OIDC linking Plan & billing Errors Status & not-stored Source

Tiny opt-in service that fronts api.ohd.dev. It is not a health-data store — events never touch this service. Its single responsibility is the profile / plan / billing triangle:

RelationCardinality
profile_ulid ↔ OIDC link1 ↔ N
profile_ulid ↔ plan1 ↔ 1
profile_ulid ↔ payment1 ↔ N
profile_ulid ↔ recovery-code hash1 ↔ 1

Authentication

Bearer tokens are HS256-signed JWTs scoped to a single profile: 

{ "sub": "<profile_ulid>", "iat": ..., "exp": ..., "iss": "ohd-saas" }

Tokens are minted at POST /v1/account, POST /v1/account/recover, and POST /v1/account/oidc/claim. They're valid for 90 days; refresh by re-claiming via recovery code or OIDC. Pass as Authorization: Bearer <token> on protected calls.

Account

POST /v1/account no auth

Register or claim a profile. Idempotent on (profile_ulid, recovery_code); conflict (409) if profile_ulid is already known with a different recovery hash.

Body

{
  "profile_ulid": "01HMYQK2BNW3D9C5XYZ4ABCDEF",
  "recovery_code": "ABCD EFGH IJKL MNOP …"   // 16x8 Crockford-base32, whitespace tolerant
}

Response (200)

{
  "profile_ulid": "01HMYQK2BNW3D9C5XYZ4ABCDEF",
  "plan": "free",
  "access_token": "<jwt>",
  "created_at": "2026-05-13T00:00:00Z"
}
GET /v1/account/me bearer

Current profile + plan + linked OIDC identities.

Response (200)

{
  "profile_ulid": "01HMYQK2BNW3D9C5XYZ4ABCDEF",
  "plan": "free",
  "created_at": "2026-05-13T00:00:00Z",
  "linked_identities": [
    { "provider": "https://accounts.google.com", "sub": "1234", "display_label": "…", "linked_at": "…" }
  ],
  "plan_info": {
    "plan": "free",
    "retention_days": 7,
    "max_storage_mb": 25,
    "sync": false
  }
}

Recovery

POST /v1/account/recover no auth

Submit a recovery code, receive a new access token. The code is canonicalised before hashing — whitespace, hyphens, and case are ignored, so "abcd-efgh-ijkl-mnop" matches "ABCD EFGH IJKL MNOP".

Body

{ "recovery_code": "ABCD EFGH IJKL MNOP …" }

Response (200)

{
  "profile_ulid": "01HMYQK2BNW3D9C5XYZ4ABCDEF",
  "plan": "free",
  "access_token": "<jwt>",
  "created_at": "2026-05-13T00:00:00Z"
}

404 if no profile matches the canonical hash.

OIDC linking

POST /v1/account/oidc/link bearer

Link an OIDC identity to the current profile. Idempotent on (provider, sub); 409 if the pair is already bound to a different profile.

Body

{
  "provider": "https://accounts.google.com",
  "sub": "1234567890",
  "display_label": "jakub@example.com"   // optional
}
DELETE /v1/account/oidc bearer

Unlink a (provider, sub) pair from the current profile.

Query

NameTypeRequired
providerstringyes
substringyes
POST /v1/account/oidc/claim no auth

"Already have an account?" — find a profile by OIDC sub and receive its access token. Used when the user wipes the device and wants to recover via Google / Apple / Email instead of typing the recovery code.

Body

{ "provider": "https://accounts.google.com", "sub": "1234567890" }

Plan & billing

GET /v1/account/plan bearer

Plan limits as a flat shape so clients can render the upsell without hard-coding numbers.

Response (200)

{
  "plan": "free",
  "retention_days": 7,           // -1 = unlimited (Paid)
  "max_storage_mb": 25,
  "sync": false
}
POST /v1/account/plan/checkout bearer

Stripe stub today — returns a placeholder URL the app opens in the browser.

Response (200)

{ "checkout_url": "https://ohd.dev/roadmap.html#payments", "stub": true }

Errors

Every error has the same shape:

{ "error": "<message>" }
StatusMeaning
400Bad request — missing or malformed field
401Missing / invalid bearer token
404Profile / link / payment not found
409Conflict — `profile_ulid` reused with wrong recovery hash, or OIDC pair bound to a different profile
500Internal — DB or signing failure. Logged server-side.

Status & what we deliberately don't store

The service is intentionally tiny. Recovery codes are stored only as SHA-256 hashes (canonicalised: case/whitespace/hyphen-tolerant). OIDC tokens are not stored — only the issuer + opaque sub. Names, addresses, emails live at the OIDC provider unless explicitly stored on a payment record for invoicing.

NOT storedWhy
Health eventsPeriod. They live in your OHD storage instance.
Plaintext recovery codesOnly SHA-256 hash kept.
OIDC ID tokensWe only need (provider, sub) to recognise you.
Names / addresses / emailsOIDC provider has them already.
